Finite State Machine

(Gambar 2) FSM sender RDT 3.0"

Apa itu Finite State Machine ?

FSM atau Finite State Machine adalah sebuah model komputasi yang digunakan untuk menggambarkan logika secara sekuensial, atau untuk menunjukkan dan kontrol alur eksekusi. Model komputasi FSM dibuat berdasarkan hypotical mesin yang membuat satu atau lebih state. Hanya satu (single) bit yang aktif pada mesin ini, sehingga mesin harus berubah transisi dari state satu ke state yang lain untuk melakukan kegiatan yang berbeda.

FSM kadang disebut juga dengan FSA (Finite State Automata) berdasarkan Setiawan FSM adalah sebuah metodologi perancangan sistem kontrol yang menggambarkan tingkah laku atau prinsip kerja sistem dengan menggunakan tiga hal berikut: State (Keadaan), Event (kejadian) dan action (aksi). Pada satu saat dalam periode waktu yang cukup signifikan, sistem akan berada pada salah satu state yang aktif. Sistem dapat beralih atau bertransisi menuju state lain jika mendapatkan masukan atau event tertentu, baik yang berasal dari perangkat luar atau komponen dalam sistemnya itu sendiri. Transisi keadaan ini umumnya juga disertai oleh aksi yang dilakukan oleh sistem ketika menanggapi masukan yang terjadi. Aksi yang dilakukan tersebut dapat berupa aksi yang sederhana atau melibatkan rangkaian proses yang relatif kompleks.

Diagram yang ada diatas memperlihatkan FSM dengan dua buah state dan dua buah input serta empat buah aksi output yang berbeda, seperti terlihat pada gambar, ketika sistem mulai dihidupkan, sistem akan bertransisi menuju state0, pada keadaan ini sistem akan menghasilkan Action1 jika terjadi masukan Event0, sedangkan jika terjadi Event1 maka Action2 akan dieksekusi kemudian sistem selanjutnya bertransisi ke keadaan State1 dan seterusnya.

Dalam bahasa pemrograman prosedural seperti bahasa C, FSM ini umumnya direalisasikan dengan menggunakan statemen kontrol switch case atau/dan if..then. Dengan menggunakan statemen-statemen kontrol ini, aliran program secara praktis akan mudah dipahami dan dilacak jika terjadi kesalahan logika. Contoh penggunaan FSM dalam dunia nyata. Lampu lalu lintas. Yang disebut State adalah lampu merah, lampu kuning, lampu hijau dan Transisi adalah setelah melewati waktu yang telah ditetapkan, lampu merah akan berubah menjadi lampu kuning, lampu kuning akan berubah menjadi lampu hijau, dan lampu hijau akan berubah menjadi lampu merah.

Kelebihan dan Kekurangan FSM (Menurut Brownlee) :
Kelebihan
  • Sederhana, sehingga mudah diimplementasikan
  • Bisa diprediksi responnya
  • Komputasi ringan
  • Relatif fleksibel
  • Merupakan metode AI lama yang bisa digunakan pada berbagai sistem
  • Mudah ditransfer dari abstrak menjadi kode program
Kekurangan
  • Karena sifatnya bisa diprediksi, maka implementasi pada game kurang disukai
  • Implementasi pada sistem yang lebih besar lebih sulit karena pengaturan dan pemeliharaannya jadi kompleks
  • Sebaiknya hanya digunakan pada sistem dimana sifat sistem bisa didekomposisi menjadi state.

Prinsip kerja FSM pada sender side

(Gambar 2) FSM sender RDT 3.0"

Ketika proses mengirimkan data terkadang kita mendapatkan kesalahan, seperti bit error atau packet loss. Dengan RDT 3.0 kesalahan tersebut dapat ditangani dengan menggunakan checksum, sequence number, ACK packets, dan retransmission. Beberapa tanda penting yang harus diperhatikan adalah :

  • Tanda panah putus-putus : Menunjukkan state awal FSM (inisialisasi)
  • Tanda panah biru : Proses transisi dari state satu ke state lain (termasuk satu state kembali ke state asalnya).
  • Garis horizontal : Menandakan event dan action pada state tertentu. Contoh event adalah rdt_send(data), timeout, rdt_rcv(rcvpkt) && (corrupt(rcvpkt) || isACK(rcvpkt, 0)), dsb. Contoh action adalah Ʌ, udt_send(sendpkt), stop_timer, dsb.
  • Simbol ( Ʌ ) : Menandakan bahwa tidak ada action untuk event tersebut.

Penjelasan tiap state pada Sender Side RDT 3.0 : ( Klik nama state)

Wait for call 0 from above

Pada state ini data dari application layer akan di kirim ke transport layer ditandai dengan 2 event yaitu :
  1. rdt_send(data), event ini berarti data telah diterima transport layer kemudian akan dilanjut dengan action di dalam event ini:
    • sndpkt = make_pkt(0, data, checksum) : Ubah data ke dalam bentuk paket dan tambahkan informasi berupa ACK 0, data, kemudian checksum.
    • udt_send(sndpkt) : Kirim paket ke network layer
    • start_timer : Mulai countdown
  2. rdt_rcv(rcvpkt), untuk event ini akan dimulai ketika state wait for ACK 1 telah dijalankan. Event ini berarti bahwa reveiver menerima paket dan ACK yang diterima sesuai dengan yang dikirim sender.
    Pada event ini tidak ada action apapun ( Ʌ ).

Wait for ACK 0

Pada state ini ditandai dengan 3 event yaitu
  1. rdt_rcv(rcvpkt) && ( corrupt(rcvpkt)) || isACK(rcvpkt, 1) ), event ini berarti bahwa sender side menerima ack 1 walaupun state aktif sekarang wait for ACK 0, hal ini menandakan bahwa paket atau ACK yang diterima oleh receiver tercorrupt.
    Pada event ini tidak ada action apapun ( Ʌ ).
    Kemudian state akan kembali menjadi wait for ACK 0 dan menunggu hingga event selanjutnya (timeout).
  2. timeout, event ini berarti waktu countdown yang dimulai pada state sebelumnya telah habis. Karena ACK 0 belum diterima oleh sender, kemudian dilanjut dengan action pada event ini :
    • udt_send(sndpkt) : Mengirimkan ulang paket
    • start_timer : Mulai countdown timer
    Kemudian state akan kembali menjadi wait for ACK 0 dan menunggu hingga event selanjutnya (rdt_rcv(rcvpkt) && notcorrupt(rcvpkt) && isACK(rcvpkt, 0) ).
  3. rdt_rcv(rcvpkt) && notcorrupt(rcvpkt) && isACK(rcvpkt, 0), event ini berarti paket dengan ACK 0 diterima dan tidak ada data yang tercorrupt, sehingga dilanjut dengan action pada event ini :
    • stop_timer : Matikan countdown.

Wait for call 1 from above

Pada state ini ditandai dengan 2 event :
  1. rdt_rcv(rcvpkt), event ini berarti bahwa reveiver menerima paket dan ACK yang diterima sesuai dengan yang dikirim sender.
    Pada event ini tidak ada action apapun ( Ʌ ).
  2. rdt_send(data), event ini berarti data telah diterima transport layer sender, kemudian akan dilanjut dengan action di dalam event ini:
    • sndpkt = make_pkt(1, data, checksum) : Ubah data ke dalam bentuk paket dan tambahkan informasi berupa ACK 1, data, kemudian checksum.
    • udt_send(sndpkt) : Kirim paket ke network layer
    • start_timer : Mulai countdown

Wait for ACK 1

Pada state ini ditandai dengan 3 event yaitu :
  1. rdt_rcv(rcvpkt) && ( corrupt(rcvpkt)) || isACK(rcvpkt, 0) ), event ini berarti bahwa sender side menerima ack 0 walaupun state aktif sekarang wait for ACK 1, hal ini menandakan bahwa paket atau ACK yang diterima oleh receiver tercorrupt.
    Pada event ini tidak ada action apapun ( Ʌ ).
    Kemudian state akan kembali menjadi wait for ACK 1 dan menunggu hingga event selanjutnya (timeout).
  2. timeout, event ini berarti waktu countdown yang dimulai pada state sebelumnya telah habis. Karena ACK 1 belum diterima oleh sender, kemudian dilanjut dengan action pada event ini :
    • udt_send(sndpkt) : Mengirimkan ulang paket
    • start_timer : Mulai countdown timer
    Kemudian state akan kembali menjadi wait for ACK 1 dan menunggu hingga event selanjutnya (rdt_rcv(rcvpkt) && notcorrupt(rcvpkt) && isACK(rcvpkt, 1) ).
  3. rdt_rcv(rcvpkt) && notcorrupt(rcvpkt) && isACK(rcvpkt, 0), event ini berarti paket dengan ACK 1 diterima dan tidak ada data yang tercorrupt, sehingga dilanjut dengan action pada event ini :
    • stop_timer : Matikan countdown

Prinsip kerja FSM pada receiver side

(Gambar 2) FSM receiver RDT 2.2

FSM dari receiver side menggunakan RDT 2.2, perbedaan dari FSM sender dengan FSM receiver side adalah timeout. Pada FSM sender side waktu timeout mengalami penambahan.

Penjelasan tiap state pada Receiver Side RDT 2.2 : ( Klik nama state)

Wait for 0 from below

Pada state ini ditandai dengan 2 event yaitu :
  1. rdt_rcv(rcvpkt && notcorrupt(rcvpkt) && has_seq0(rcvpkt), event ini berarti bahwa paket yang dikirim sender masuk ke layer network. Paket ini tidak tercorrupt serta memiliki sequence number 0 sesuai dengan yang dikirim oleh sender. Kemudian dilanjut dengan action pada event ini :
    • extract(rcvpkt,data) : Lakukan ekstraksi pada paket, sehingga menjadi kedalam bentuk data dalam transport layer.
    • deliver_data(data) : Kirim data ke application layer
    • sndpkt=make_pkt(ACK,0,checksum): Ubah data ke dalam bentuk paket dan tambahkan informasi berupa ACK, sequence number 0 (nol), kemudian checksum.
    • udt_send(sndpkt) : Kirim paket ke network layer
  2. dt_rcv(rcvpkt) && ( corrupt(rcvpkt)||has_seq1(rcvpkt) ), untuk event ini akan dimulai ketika state wait for 1 from below telah dijalankan. Event ini akan aktif apabila paket yang diterima tercorrupt atau memiliki sequence number 1 (perhatikan bahwa sequence number yang diharapkan oleh receiver adalah 0). Kemudian akan dilanjut oleh action pada event ini :
    • sndpkt=make_pkt(ACK, 1, checksum) : Buat paket yang berisi informasi berupa ACK, sequence number 1, kemudian checksum.
    • udt_send(sndpkt) : Kirim paket ke network layer
    Kemudian state akan kembali menjadi wait for 0 from below dan menunggu hingga event selanjutnya ( rdt_rcv(rcvpkt && notcorrupt(rcvpkt) && has_seq0(rcvpkt) ).

Wait for 1 from below

State ini akan aktif apabila sequence number yang dikirim oleh sender adalah 1. Pada state ini ditandai dengan 2 event yaitu :
  1. rdt_rcv(rcvpkt) && ( corrupt(rcvpkt)|| has_seq0(rcvpkt) ), event ini akan aktif apabila paket yang diterima tercorrupt atau memiliki sequence number 0 (perhatikan bahwa sequence number yang diharapkan oleh receiver adalah 1). Sehingga akan ada satu masalah dalam pengiriman paket, paket mengalami corrupt atau paket yang dikirim duplikat. Kemudian akan dilanjut oleh action pada event dibawah ini :
    • sndpkt=make_pkt(ACK,0,checksum) : Buat paket yang berisi informasi berupa ACK, sequence number 0 (nol), kemudian checksum.
    • udt_send(sndpkt) : Kirim paket ke network layer
    Jika paket terindikasi mengalami corrupt atau duplikat maka penerima tidak akan mengekstrak paket tersebut melainkan hanya mengirimkan paket yang berisi ACK, sequence number 0 dan checksum. Kemudian state akan kembali menjadi wait for 1 from below dan menunggu hingga event selanjutnya ( rdt_rcv(rcvpkt) && notcorrupt(rcvpkt) && has_seq1(rcvpkt)).
  2. rdt_rcv(rcvpkt) && notcorrupt(rcvpkt) && has_seq1(rcvpkt), event ini berarti bahwa paket yang dikirim sender masuk ke layer network. Paket ini tidak tercorrupt serta memiliki sequence number 1 sesuai dengan yang dikirim oleh sender. Kemudian dilanjut dengan action pada event ini :
    • extract(rcvpkt,data) : Lakukan ekstraksi pada paket, sehingga menjadi kedalam bentuk data dalam transport layer
    • deliver_data(data) : Kirim data ke application layer
    • sndpkt=make_pkt(ACK,1,checksum) : Ubah data ke dalam bentuk paket dan tambahkan informasi berupa ACK, sequence number 1, kemudian checksum.
    • udt_send(sndpkt) : Kirim paket ke network layer
    Jika pada state selanjutnya paket tercorrupt atau terdapat duplikasi paket, kemudian paket tidak akan diekstrak, receiver akan mengirimkan pesan ACK, sequence number 1 yang ada dalam state wait for 0 from below pada event rdt_rcv(rcvpkt) && ( corrupt(rcvpkt)||has_seq1(rcvpkt)).
